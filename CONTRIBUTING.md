# 如何貢獻這本書

感謝你願意幫忙讓這本書更好。這是一本活文件，每一個改進都有意義。

## 回報錯誤

發現錯字、過時資訊、或哪個步驟跑不起來？

1. 開一個 [Issue](../../issues/new)
2. 標題寫清楚：例如「ch03 - 第二節的指令在 macOS 15 無法執行」
3. 說明你遇到的問題、你的環境（OS、Claude Code 版本）、期待的行為

不需要先問「可以開 Issue 嗎」，直接開就對了。

## 建議改進

有更好的說明方式、想補充實際案例、或覺得某個概念解釋得不夠清楚？

1. 先開 Issue 說明你的想法，等確認方向再動手寫 PR
2. 這樣可以避免你花了時間寫，最後發現方向不符合這本書的定位

## 寫作風格規範

這本書有固定的寫作風格，PR 請盡量符合：

**語言**
- 全書繁體中文
- 技術詞彙首次出現時附英文原文，例如：代理人（Agent）
- 中英文之間加一個空格，例如：Claude Code 很好用

**段落**
- 每個段落最多 3 行
- 一個段落說一件事，不要塞太多概念進去
- 用「我」，不用「我們」（除非真的是集體的事）

**語氣**
- 禁止使用：絕對、一定、最好的、唯一、完美
- 鼓勵誠實描述限制：「Claude 有時候會...」、「這個方法不適合...」
- Permission-based 語氣：「如果你覺得可以，試試看」

**程式碼範例**
- 所有程式碼範例必須可以執行，不能只是示意
- 指令前加 `$` 表示 terminal 輸入
- 輸出結果用獨立的 code block 呈現，不要混在一起

**內容取向**
- 實用優先，不做理論推導
- 每個概念都要有「你什麼時候會用到這個」的說明
- 如果你想補充原理，放在段落最後，不要放在最前面

## PR 流程

1. Fork 這個 repo
2. 建立一個描述性的 branch，例如：`fix/ch03-command-typo` 或 `improve/ch05-hooks-example`
3. 改好之後開 PR，說明你改了什麼、為什麼這樣改
4. 等待 review，可能會有來回討論

## 謝謝你

每一個 Issue、每一個 PR，不管大小，都讓這本書更好。
特別感謝那些願意在社群裡分享使用心得、把錯誤回報給我的人——你們是這本書持續更新的動力。
